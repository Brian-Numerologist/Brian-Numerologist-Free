VI. LEAD ID SPEC
============================================================

lead_id bắt buộc.

Format:
LEAD_YYYYMMDD_HHMMSS_RANDOM

Ví dụ:
LEAD_20260615_213015_X7K2

Quy tắc:
- Sinh ở frontend trước khi gửi.
- Nếu chưa có lead_id trong localStorage, tạo mới.
- Nếu cùng session khách bấm nhiều hành động, giữ cùng lead_id.
- Nếu khách quay lại sau và localStorage còn, có thể dùng lại lead_id.
- Nếu khách dùng thiết bị khác, tạo lead mới.

Random:
- 4–6 ký tự A-Z/0-9.

============================================================
VII. DUPLICATE HANDLING
============================================================

Brian chọn:
- Luôn tạo dòng mới.
- Có duplicate_warning.

MVP không cần tìm và cập nhật dòng cũ.

duplicate_warning có thể là:
- empty
- same_phone_possible_duplicate
- same_name_birthdate_possible_duplicate
- same_phone_and_name_birthdate_possible_duplicate

Cách làm đơn giản:
- Frontend chỉ gửi dữ liệu.
- Apps Script có thể kiểm tra những dòng gần nhất trong leads:
  + Nếu phone_or_zalo trùng → duplicate_warning = same_phone_possible_duplicate.
  + Nếu full_name_original + birth_date trùng → duplicate_warning = same_name_birthdate_possible_duplicate.
  + Nếu cả hai trùng → same_phone_and_name_birthdate_possible_duplicate.
- Dù trùng vẫn append dòng mới.

VIII. CONSENT SPEC
============================================================

consent_to_contact:
- Boolean.
- Nếu khách chỉ xem FREE và không nhập SĐT/Zalo: có thể false.
- Nếu khách nhập SĐT/Zalo: nên yêu cầu tick consent.
- Nếu khách chọn gói, yêu cầu PDF hoặc yêu cầu tư vấn: bắt buộc true.

Text checkbox đề xuất:
“Tôi đồng ý để Brian-Numerologist lưu thông tin này và liên hệ lại qua Zalo/điện thoại nếu tôi yêu cầu tư vấn hoặc bản chuyên sâu.”

Validation:
- Nếu selected_package có giá trị mà consent_to_contact = false → không cho gửi yêu cầu tư vấn.
- Nếu pdf_requested = true và phone_or_zalo có nhập → yêu cầu consent_to_contact = true.
- Nếu phone_or_zalo trống nhưng khách chọn gói → yêu cầu nhập phone_or_zalo và tick consent.

IX. ACTION TRIGGERS
============================================================

Website cần gửi/cập nhật lead theo các action:

1. Generate report

- Cập nhật status = pdf_requested.
- Gửi event vào Google Sheet.
- Gọi window.print().

4. Package click
Trigger:
- Khách bấm “Tôi quan tâm” ở một gói.

Action:
- Ghi selected_package.
- Nếu thiếu phone_or_zalo, hiện ô nhập Zalo.

- Cập nhật status = paid_report_requested.

- Hiện CTA: “Brian đã ghi nhận yêu cầu. Anh/chị có thể nhắn Zalo 0948909983 để được tư vấn nhanh hơn.”

5. Admin retry sync
Trigger:
- Brian vào admin.html và bấm “Đồng bộ lại lead lỗi”.

Action:
- Lấy pending leads từ localStorage.
- Gửi lại Apps Script.
- Nếu thành công, xóa khỏi pending queue.
- Ghi log sync_success hoặc sync_failed.

XI. GOOGLE APPS SCRIPT SPEC
============================================================

Tạo file:
google-apps-script/Code.gs

Apps Script cần có:

1. doPost(e)
- Nhận request POST từ website.
- Parse JSON.
- Kiểm tra shared_secret.
- Validate payload tối thiểu.
- Append row vào tab leads.
- Append row vào tab logs.
- Trả JSON response.

2. getSheet_(sheetName)
- Lấy sheet theo tên.
- Nếu chưa có thì tạo.

3. setupSheets()
- Tạo tab leads, config, logs.
- Tạo header nếu chưa có.

4. appendLead_(payload)
- Ghi vào tab leads.

5. appendLog_(payload, message)
- Ghi vào tab logs.

6. detectDuplicate_(lead)
- Kiểm tra trùng đơn giản.
- Vẫn append dòng mới.

7. jsonResponse_(data)
- Trả response JSON.

8. validateSecret_(payload)
- So sánh shared_secret với Script Properties.

9. setSecretOnce()
- Hàm để Brian/lập trình viên set shared_secret trong Apps Script Properties.

Apps Script response thành công:
{
  "ok": true,
  "message": "Lead saved",
  "lead_id": "LEAD_...",
  "duplicate_warning": ""
}

Apps Script response lỗi:
{
  "ok": false,
  "message": "Invalid shared secret"
}

============================================================
XII. SHARED SECRET SPEC
============================================================

Vì website public nên endpoint Apps Script có thể bị người khác thấy trong DevTools.

Dùng shared_secret chỉ là lớp bảo vệ cơ bản, không phải bảo mật tuyệt đối.

Cách dùng:
- Frontend có config placeholder.
- Apps Script có SECRET lưu trong Script Properties.
- Request gửi shared_secret.
- Apps Script kiểm tra trước khi ghi Sheet.

Tên config:
GOOGLE_SCRIPT_SHARED_SECRET

Lưu ý:
- Không commit secret thật vào GitHub public nếu muốn an toàn hơn.
- Với GitHub Pages public, secret đặt trong JS vẫn có thể bị xem.
- Mục đích của shared_secret ở Phase 2 chỉ là giảm ghi rác vô tình, không chống được attacker nghiêm túc.
- Nếu cần bảo mật thật, Phase 3 nên có backend riêng.

XIII. FRONTEND UPDATE SPEC
============================================================

Cập nhật app.js:

Các hàm mới/cần sửa:

1. generateLeadId()
- Tạo LEAD_YYYYMMDD_HHMMSS_RANDOM.

2. getOrCreateLeadId()
- Lấy lead_id hiện tại từ localStorage.
- Nếu chưa có thì tạo.

3. buildLeadPayload(eventType, status)
- Tạo payload theo schema.

4. saveLeadToGoogleSheet(payload)
- POST đến Apps Script Web App URL.
- Nếu thành công, ghi log local.
- Nếu thất bại, lưu pending queue.

5. saveLead(payload)
- Wrapper.
- Gọi saveLeadToGoogleSheet().
- Nếu lỗi, fallback localStorage.
